# Масштабирование серверной части

## Выбор языка и фреймворка

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Node.js + Socket.io** | Один язык с клиентом, удобно, большое комьюнити | Один поток (CPU-bound проблемы) |
| **Go** | Высокая производительность, горутины, низкое потребление памяти | Другой язык для изучения |
| **Python (FastAPI)** | Простой, async, быстрая разработка | Медленее, не для высоких нагрузок |
| **Rust** | Максимальная скорость, безопасность | Сложно, долгая разработка |

**Рекомендация для старта:** Node.js (ты уже на нём для клиента). Потом перейти на Go если нагрузка будет >10k онлайн.

## Архитектура под нагрузку

### Базовая схема

```
                                    ┌──────────────┐
                                    │    Nginx     │ (балансировщик, терминация SSL)
                                    └──────┬───────┘
                                           │
                    ┌──────────────────────┼──────────────────────┐
                    │                      │                      │
            ┌───────▼───────┐      ┌───────▼───────┐      ┌───────▼───────┐
            │   Game S1     │      │   Game S2     │      │   Game S3     │
            │   (Node/Go)   │      │   (Node/Go)   │      │   (Node/Go)   │
            └───────┬───────┘      └───────┬───────┘      └───────┬───────┘
                    │                      │                      │
                    └──────────────────────┼──────────────────────┘
                                           │
                                    ┌──────▼───────┐
                                    │    Redis     │ (pub/sub, кеш, сессии)
                                    └──────┬───────┘
                                           │
                                    ┌──────▼───────┐
                                    │   PostgreSQL │ (персистентность)
                                    └──────────────┘
```

### Компоненты

1. **Nginx** — балансировщик
   - Round-robin или least_conn
   - Sticky sessions (ip_hash) для WebSocket
   - SSL termination

2. **Game Servers** — серверы боя
   - Каждый сервер держит активные бои в памяти
   - Stateless для матчмейкинга

3. **Redis** — коммуникация и кеш
   - Pub/Sub между серверами
   - Хранение сессий
   - Кеш данных игроков

4. **PostgreSQL** — персистентность
   - Профили игроков
   - История матчей
   - Транзакции

## Особенности карточной игры

### Нагрузка боя

**Бой — это кратковременная сессия:**
- Длительность: 1-5 минут
- Состояние: 50-100 КБ (колода, HP, баффы)
- После боя: запись в БД, очистка памяти

**Ресурсы:**
| Параметр | Значение |
|----------|----------|
| CPU на бой (Node.js) | ~2-5% |
| CPU на бой (Go) | ~1% |
| RAM на бой | ~50KB |
| Боев на 1 сервер (Node.js) | ~100-200 |
| Боев на 1 сервер (Go) | ~500+ |

### AI врагов — не нагрузка

- Простая логика (враг атакует каждый ход)
- Вычисляется мгновенно
- Можно даже на клиенте (с верификацией на сервере)

## Масштабирование по уровням

### Уровень 1: 100-1000 игроков

```
1 сервер приложений (2 CPU, 4 GB RAM)
1 сервер Redis (1 GB RAM)
1 сервер PostgreSQL (SSD)

Стоимость: ~$20-40/месяц (DigitalOcean, AWS t3.medium)

Ограничения:
- Все бои на одном сервере
- Один PoP (point of presence)
```

### Уровень 2: 1000-10000 игроков

```
3-5 серверов приложений
1 сервер Redis (или Redis Cluster)
1 сервер PostgreSQL (или read replica)

Стоимость: ~$80-150/месяц

Оптимизации:
- Nginx балансировка
- Redis для сессий
- Кеширование профилей игроков
```

### Уровень 3: 10000+ игроков

```
10+ серверов приложений (Kubernetes)
Redis Cluster (3+ ноды)
PostgreSQL Cluster (master + read replicas)
CDN для статики

Стоимость: ~$300-1000/месяц

Особенности:
- Автоматическое масштабирование (HPA)
- Географическое распределение
- Мониторинг и алерты
- Микросервисы: auth, game, matchmaker, payment
```

## Конкретная конфигурация: 4 CPU / 8 GB RAM / 120 GB SSD

**Стек:** Node.js (без Go)

### Характеристики

| Параметр | Значение |
|----------|----------|
| CPU | 4 ядра (Node.js однопоточный, использовать cluster или 4 воркера) |
| RAM | 8 GB |
| Диск | 120 GB SSD |
| Язык | Node.js |

### Оценка производительности

#### CPU
- 1 бой ~2-5% CPU (Node.js)
- 1 воркер держит ~20-50 боёв
- 4 воркера → **80-200 параллельных боёв**

#### RAM
- Node.js процесс (база): ~150-200 MB
- 4 воркера: ~800 MB
- 1 бой в памяти: ~50-100 KB
- Доступно: 8 GB - 800 MB = 7.2 GB

**Теоретический максимум боёв:** ~70,000-140,000
**Реально (с запасом, ОС, Redis):** **100-150 боёв**

#### Диск
- SSD 120 GB достаточно для локальной БД, логов, кеша
- Для продакшена рекомендуется PostgreSQL на отдельном сервере

### Итоговая ёмкость

| Метрика | Значение |
|---------|----------|
| **Параллельных боёв** | 100-200 |
| **Одновременных игроков** | 100-200 |
| **Боёв в секунду (пик)** | ~20-40 |
| **Время отклика** | <50ms |

### Рекомендация по использованию

- **До 100 игроков онлайн:** 1 сервер — достаточно
- **100-500 игроков:** 2 сервера + Redis + PostgreSQL на отдельном сервере
- **500+ игроков:** кластер серверов + балансировщик

### Сравнение с Go

| Параметр | Node.js | Go |
|----------|---------|-----|
| Боёв на сервер | 100-200 | 500+ |
| Потребление RAM | Выше | Ниже |
| Сложность разработки | Проще | Средне |
| Однопоточный | Да (нужен cluster) | Нет |

## Оптимизация: Client-side compute + Server-side verification

### Концепция

Клиент играет бой локально (как сейчас), сервер только проверяет результат. Это значительно снижает нагрузку.

```
Этап 1: Старт боя
─────────────────────────────────────────
Сервер → Клиент:
{
  seed: "abc123",              // Уникальный ключ для хеша
  enemyId: "03_elves",
  deck: ["01_spearman", "02_militia", ...],
  shuffledDeckHash: "sha256..."  // Хеш перетасованной колоды
}

Этап 2: Игра (клиент)
─────────────────────────────────────────
Клиент играет бой локально, формирует историю ходов:
[
  { turn: 1, cards: ["01_spearman", "02_militia"], damage: 5 },
  { turn: 2, cards: ["03_prince"], damage: 5 },
  ...
]

Этап 3: Завершение
─────────────────────────────────────────
Клиент → Сервер:
{
  seed: "abc123",
  history: [...],                    // История ходов
  finalState: {
    enemyHp: 0,
    playerHp: 15,
    cardsPlayed: 6,
    reward: { type: "card", id: "04_berserk" }
  },
  hash: SHA256(seed + JSON.stringify(history))
}

Этап 4: Верификация (сервер)
─────────────────────────────────────────
Сервер:
1. Проверяет хеш
2. Воспроизводит бой (быстро, без анимаций)
3. Сравнивает результат с finalState
4. Если совпадает → начисляет награду
5. Если нет → отклоняет, логирует (возможный читер)
```

### Выигрыш в нагрузке

| Метрика | Сервер обсчитывает | Клиент + верификация |
|---------|-------------------|---------------------|
| Запросов на бой | 10-20 (каждый ход) | 2 (старт + финиш) |
| CPU на бой | ~5% | ~0.5% (только валидация) |
| RAM на бой | 50-100 KB | 0 (не храним) |
| Время отклика | >100ms | <20ms (игра локально) |
| Боёв на сервер | 100-200 | **500-1000** |

### Экономия для конфигурации 4 CPU / 8 GB

| Параметр | До | После |
|----------|-----|-------|
| Боёв параллельно | 100-200 | **500-1000** |
| Игроков онлайн | 100-200 | **200-500** |
| Боёв в секунду | 20-40 | **200-400** |

### Что проверять на сервере

1. **Хеш истории** — подделана ли история
2. **Seed** — сервер выдаёт, клиент возвращает
3. **Воспроизведение** — сервер быстро проигрывает бой и сверяет результат
4. **Начальное состояние** — не подменены ли карты/враг
5. **Лимиты** — не превышены ли ходы/сбросы

### Безопасность

- **Seed от сервера** — клиент не может подделать
- **Хеш истории** — нельзя изменить ходы задним числом
- **Воспроизведение** — сервер сам считает результат
- **Для PvP не подойдёт** — нужен real-time, но для PVE (игрок vs AI) — отлично

### Когда использовать

- ✅ PVE (игрок vs AI)
- ✅ Пошаговые игры
- ✅ Игры с длинными сессиями

- ❌ PvP (real-time)
- ❌ Игры с высокой конкурентностью

## Резюме

1. **Начать с Node.js** — проще, один язык с клиентом
2. **Redis + PostgreSQL** — стандартный стек
3. **Масштабировать по мере роста** — начинать с одного сервера
4. **Go — когда потребуется** — >10k игроков или высокие требования к latency
5. **Мониторинг сразу** — Prometheus + Grafana
